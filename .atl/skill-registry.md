## Compact Rules (for sub-agent injection)

### Stack & language
- Node.js + Express + Sequelize + PostgreSQL, CommonJS, pnpm.
- Domain naming is **Spanish** (Trabajador, Cliente, Ascensor, Programacion, Mantenimiento, Ruta, etc.). Match the existing vocabulary — do not Anglicize.

### Architecture
- Layering: `routes → middlewares (authenticate → authorize → validate) → controllers → services → models`.
- Controllers are thin: parse req, delegate to service, return via `utils/response.util`.
- All business logic + Sequelize queries live in `src/services/`.
- **Associations are centralized in `src/models/index.js`** — never define associations inside individual model files. Import models from `require('../models')`.
- Adding a resource = `models/X.js` + association in `models/index.js` + `validators/X.validator.js` + `services/X.service.js` + `controllers/X.controller.js` + `routes/Xroutes.js` + register router in `src/app.js`.

### Auth & authorization
- JWT. Token comes from `accessToken` cookie (preferred) or `Authorization: Bearer …` header. `req.user = { id, dni, correo, rol, tipo }`.
- Roles: `ADMIN`, `TECNICO`. Apply `authenticate` then `authorize('ADMIN')` on protected routers.
- Refresh tokens are server-side rows in `Sesion` — logout invalidates them.
- CORS allow-list is hard-coded in `src/app.js`; add new origins there (not just via `CORS_ORIGIN`).

### Validation & responses
- Joi schemas in `src/validators/`, enforced via `validate(schema)` middleware.
- Use `successResponse`/`errorResponse` from `src/utils/response.util.js`. Error shape: `{ success: false, message, error: <CODE>, timestamp }` where `<CODE>` is a machine code like `TOKEN_EXPIRED`, `NO_TOKEN`, `API_NOT_FOUND`.

### Database & migrations
- Schema changes live as **raw SQL** in `scripts/migrations/` (NOT Sequelize migrations). Always pair the SQL with model updates.
- `DB_SYNC=true` triggers `sequelize.sync({ alter: true })` on boot — off by default in production.
- Two entry points exist (`src/server.js` for local, `src/app.js` for Render direct). Keep startup logic mirrored.
- `process.env.TZ = 'America/Lima'` is set at the top of `src/app.js` and is load-bearing for scheduling math.

### IA-Scheduler subsystem (`src/services/ia-scheduler/`)
- 2-layer: deterministic motor (`motor.service.js`, `worker.service.js`, `demand.service.js`, `district-times.service.js`) + LLM adjuster (`llm.service.js`, Anthropic Claude, default `claude-haiku-4-5`).
- **Hard rules live in code, never in the prompt.** The LLM may reorder/exclude alternatives but never invent slots or violate the eligibility matrix.
- Strip markdown fences before `JSON.parse` on Claude responses (Claude sometimes wraps in ```json … ```).
- On LLM failure, fall back to the raw motor evaluation — never block the admin.
- Services are lazily instantiated (module-level singletons) — importing the controller must not hit the DB.
- Configuration tables: `ConfiguracionIA` (knobs) and `TablaDistritoLima` (travel matrix).

### Programacion model quirk
- A `Programacion` references up to 4 technicians via 4 separate columns (`trabajador_id`, `tecnico2_id`, `tecnico3_id`, `tecnico4_id`) with aliases `Tecnico1`–`Tecnico4` in `models/index.js`.
- "All jobs for technician X today" queries must OR across all four columns (see `WorkerService._obtenerDatosDelDia`).
- Do not refactor to a join table without coordination.

### Testing
- Jest 30 + supertest. Test paths: `tests/**/*.test.js` and `src/**/__tests__/**/*.test.js`.
- `pnpm test` runs `jest --watchAll` by default — pass `--watchAll=false` for CI / one-shot runs.
- `jest.config.js` sets `forceExit: true` and auto-clears/resets/restores mocks between tests.
- No linter, no type checker, no formatter — do not assume any of these exist.

### Deployment
- Render (oregon, free plan). Build: `npm install -g pnpm && pnpm install --frozen-lockfile`. Start: `pnpm start`. Health check: `GET /`.
- DB credentials injected from Render's managed Postgres. `DB_SSL=true` only when not on localhost.
